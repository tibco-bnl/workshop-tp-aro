# Release Notes - v1.17.0

**Release Date:** May 18, 2026  
**Last Updated:** May 18, 2026  
**TIBCO Platform Control Plane Version:** 1.17.0

## Overview

This release contains documentation aligned with **TIBCO Platform Control Plane version 1.17.0** for deploying TIBCO Platform on Azure Red Hat OpenShift (ARO) clusters.

## What's New in 1.17.0

> **Source:** [Official TIBCO Platform 1.17.0 Release Notes](https://docs.tibco.com/pub/platform-cp/latest/doc/html/Default.htm#Release-Notes/new-features.htm)

### TIBCO Control Plane New Features

#### Webhook Receiver for Alerts
- **Standardized JSON Payload**: Integrate alert notifications with external systems using a standardized JSON payload
- **External System Integration**: Connect alerts to third-party monitoring platforms, ticketing systems, or custom notification workflows
- **Configurable Endpoints**: Define custom webhook endpoints per alert rule or globally
- For more information, see [Configuring Webhook Notification Receiver](https://docs.tibco.com/pub/platform-cp/latest/doc/html/UserGuide/configuring-webhook-receiver.htm)

#### OpenSearch Support for Observability
- **Jaeger Traces Integration**: Use OpenSearch as the backend for distributed tracing with Jaeger
- **Service Log Storage**: Store and query service logs in OpenSearch indices
- **Flexible Observability Stack**: Greater flexibility to choose between Elasticsearch and OpenSearch for your observability infrastructure
- **OpenSearch Index Templates**: Optimized index template management for TIBCO Platform workloads
- For more information, see [OpenSearch Index Templates](https://docs.tibco.com/pub/platform-cp/latest/doc/html/UserGuide/jaeger-opensearch-index-template.htm)

#### Capability Management APIs
- **Capability Update API**: New PUT method API to update existing capability instances without re-provisioning
- **Platform API to Upgrade Capability Instances**: New API endpoint to upgrade capability instances programmatically, simplifying automated upgrade workflows

### TIBCO BusinessWorks™ 6 (Containers)

#### Custom Fluentbit Configuration
- **User-Defined Fluentbit Config**: Set customized Fluentbit configurations while provisioning or updating the BW6 (Containers) Capability
- **Log Pipeline Customization**: Tailor log parsing, filtering, and routing rules to meet operational requirements
- **Supported from**: Capability version 1.17.0 and above

## Upgrade Path

### From v1.16.0 to v1.17.0

TIBCO provides documented upgrade procedures. Follow these general steps:

1. **Backup Current Installation**:
   ```bash
   # Backup all helm releases
   for release in $(helm list -n cp1-ns -o json | jq -r '.[].name'); do
     helm get values $release -n cp1-ns > ${release}-values-backup.yaml
   done
   
   # Backup OpenShift routes
   oc get routes -n cp1-ns -o yaml > routes-backup.yaml
   ```

2. **Update Container Registry Credentials** (if registry changed):
   ```bash
   # Verify current registry in use
   oc get secret tibco-container-registry-credentials -n cp1-ns -o jsonpath='{.data.\.dockerconfigjson}' | base64 -d | jq '.auths | keys'
   ```

3. **Update Helm Repository**:
   ```bash
   helm repo update tibco-platform
   ```

4. **Upgrade Control Plane Components**:
   ```bash
   # Upgrade tibco-cp-base
   helm upgrade tibco-cp-base tibco-platform/tibco-cp-base \
     --version 1.17.0 \
     -n cp1-ns \
     -f cp1-values.yaml
   ```

5. **Configure OpenSearch (if using OpenSearch)**:
   ```bash
   # Apply OpenSearch index templates
   # See: https://docs.tibco.com/pub/platform-cp/latest/doc/html/UserGuide/jaeger-opensearch-index-template.htm
   ```

6. **Verify Upgrade**:
   ```bash
   # Check all pods are running
   oc get pods -n cp1-ns
   
   # Check helm releases
   helm list -n cp1-ns
   
   # Verify routes
   oc get routes -n cp1-ns
   ```

### From v1.15.0 to v1.17.0

**Not Recommended**: Direct upgrade from v1.15.0 to v1.17.0 is not supported.

**Recommended Path**: Upgrade to v1.16.0 first, then upgrade to v1.17.0:
1. Follow [v1.15.0 to v1.16.0 upgrade guide](./v1.16.0#upgrade-path)
2. Then follow v1.16.0 to v1.17.0 upgrade steps above

## Known Issues & Workarounds

### ARO-Specific Considerations

1. **OpenSearch Route Configuration**:
   - **Note**: If using OpenSearch for observability, configure the OpenShift Route with appropriate TLS termination (edge or passthrough depending on your OpenSearch TLS config)
   - **Recommendation**: Use `reencrypt` termination for end-to-end TLS with OpenSearch

2. **Azure Files Storage Class**:
   - **Issue**: Azure Files may have slower performance for database workloads
   - **Workaround**: Use Azure Disk (Premium SSD) for PostgreSQL databases

3. **Security Context Constraints**:
   - **Issue**: Some TIBCO components may require custom SCC
   - **Workaround**: Apply appropriate SCCs as documented in setup guides

4. **BW5 Hawk REST API Port**:
   - **Note**: The new Hawk REST API on port 8090 requires a network policy update if you have restrictive egress/ingress rules for BW5CE pods
   - **Workaround**: Add port 8090 to your NetworkPolicy for BW5CE pods

5. **Webhook Receiver Firewall Rules**:
   - **Note**: If using the new Webhook Receiver for Alerts, ensure your Control Plane namespace can reach the external webhook endpoint
   - **Workaround**: Update NSG rules and OpenShift egress policies as required

## Getting Started

### Quick Start for ARO

1. **Prerequisites**:
   - Azure Red Hat OpenShift cluster (4.14+)
   - `oc` CLI configured and connected to ARO cluster
   - Helm 3.13+ installed
   - Access to TIBCO container registry

2. **Setup Guides**:
   - [Control Plane + Data Plane Setup Guide](../howto/how-to-cp-and-dp-openshift-aro-aks-setup-guide.md)
   - [Data Plane Only Setup Guide](../howto/how-to-dp-openshift-aro-aks-setup-guide.md)

3. **Key Differences from AKS**:
   - Uses OpenShift Routes instead of Kubernetes Ingress
   - Requires Security Context Constraints (SCC)
   - Different storage class naming (Azure Files, Azure Disk)
   - Uses `oc` CLI instead of `kubectl`
