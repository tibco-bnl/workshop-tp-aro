## Upgrade Path

### From v1.15.0 to v1.16.0

TIBCO provides documented upgrade procedures. Follow these general steps:

1. **Backup Current Installation**:
   ```bash
   # Backup all helm releases
   for release in $(helm list -n cp1-ns -o json | jq -r '.[].name'); do
     helm get values $release -n cp1-ns > ${release}-values-backup.yaml
   done
   
   # Backup OpenShift routes
   oc get routes -n cp1-ns -o yaml > routes-backup.yaml
   ```

2. **Update Container Registry Credentials**:
   ```bash
   # Delete old secret
   oc delete secret tibco-container-registry-credentials -n cp1-ns
   
   # Create new secret with updated registry
   oc create secret docker-registry tibco-container-registry-credentials \
     -n cp1-ns \
     --docker-server=csgprdusw2reposaas.jfrog.io \
     --docker-username=tibco-platform-devqa \
     --docker-password="${TP_CONTAINER_REGISTRY_PASSWORD}"
   ```

3. **Update Helm Repository**:
   ```bash
   helm repo update tibco-platform
   ```

4. **Upgrade Control Plane Components**:
   ```bash
   # Upgrade tibco-cp-base
   helm upgrade tibco-cp-base tibco-platform/tibco-cp-base \
     --version 1.16.0 \
     -n cp1-ns \
     -f cp1-values.yaml
   ```

5. **Verify Upgrade**:
   ```bash
   # Check all pods are running
   oc get pods -n cp1-ns
   
   # Check helm releases
   helm list -n cp1-ns
   
   # Verify routes
   oc get routes -n cp1-ns
   ```

### From v1.14.0 to v1.16.0

**Not Recommended**: Direct upgrade from v1.14.0 to v1.16.0 is not supported.

**Recommended Path**: Upgrade to v1.15.0 first, then upgrade to v1.16.0:
1. Follow [v1.14.0 to v1.15.0 upgrade guide](./v1.15.0#upgrade-path)
2. Then follow v1.15.0 to v1.16.0 upgrade steps above

## Known Issues & Workarounds

### ARO-Specific Considerations

1. **OpenShift Router Default Routes**:
   - **Issue**: Default routes may conflict with custom domain configurations
   - **Workaround**: Use `route.openshift.io/termination` annotations correctly

2. **Azure Files Storage Class**:
   - **Issue**: Azure Files may have slower performance for database workloads
   - **Workaround**: Use Azure Disk (Premium SSD) for PostgreSQL databases

3. **Security Context Constraints**:
   - **Issue**: Some TIBCO components may require custom SCC
   - **Workaround**: Apply appropriate SCCs as documented in setup guides

4. **Network Policies with OVN**:
   - **Issue**: OVN-Kubernetes may behave differently than OpenShift SDN
   - **Workaround**: Test network policies thoroughly in OVN environments

## Getting Started

### Quick Start for ARO

1. **Prerequisites**:
   - Azure Red Hat OpenShift cluster (4.14+)
   - `oc` CLI configured and connected to ARO cluster
   - Helm 3.13+ installed
   - Access to TIBCO container registry

2. **Setup Guides**:
   - [Control Plane + Data Plane Setup Guide](../howto/v1.16/how-to-cp-and-dp-openshift-aro-aks-setup-guide.md)
   - [Data Plane Only Setup Guide](../howto/v1.16/how-to-dp-openshift-aro-aks-setup-guide.md)

3. **Key Differences from AKS**:
   - Uses OpenShift Routes instead of Kubernetes Ingress
   - Requires Security Context Constraints (SCC)
   - Different storage class naming (Azure Files, Azure Disk)
   - Uses `oc` CLI instead of `kubectl`
